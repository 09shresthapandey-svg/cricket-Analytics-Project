Cricket Analytics Project

Introduction

Cricket is one of the most widely followed sports in the world, with billions of viewers across countries such as India, England, Australia, Pakistan, and South Africa. The game is played in multiple formats including Test cricket, One Day Internationals (ODIs), and Twenty20 (T20), each demanding different skills, strategies, and performance expectations from players. Performance in cricket is influenced by a complex interaction of factors such as player roles, match format, pitch conditions, weather, opposition strength, and venue characteristics. Batting, bowling, and all-round performance are measured using a variety of statistics including runs scored, strike rate, economy rate, wickets taken, and match impact. Understanding how these variables interact is important for teams, analysts, coaches, and selectors aiming to improve performance and decision-making.

In recent years, the availability of large-scale cricket data has transformed how the sport is analyzed and understood. Advanced analytics are now used to evaluate player consistency, predict match outcomes, identify emerging talent, and optimize team composition. Organizations such as ESPN Cricinfo and Cricsheet provide ball-by-ball and match-level data that enables deeper exploration of performance patterns across formats and competitions. Prior research and analysis in cricket have examined batting order optimization, home advantage, pitch effects, and player aging curves. However, there is still significant opportunity to explore how multiple variables jointly influence performance using modern data science and machine learning techniques. This project focuses on exploring cricket data to uncover meaningful patterns, relationships, and insights that explain player and team success.

This project also applies unsupervised learning methods to cricket performance data to discover structure that is not obvious from raw tables or simple charts. In particular, Principal Component Analysis (PCA) is used to reduce dimensionality while retaining most of the variance (information), which helps us visualize player and match behavior in 2D and 3D. Clustering methods (KMeans, Hierarchical, and DBSCAN) are used to group similar performances or player profiles based on statistical similarity. Finally, Association Rule Mining (ARM) is used to identify “if-then” style relationships (rules) among events in the data (for example, combinations of match situations and outcomes). These approaches help convert complex ball-by-ball data into interpretable insights that can support selection decisions, strategy planning, and performance evaluation.

PCA (Principal Component Analysis)

What is PCA? (1 paragraph)

Principal Component Analysis (PCA) is a dimensionality reduction technique that transforms a dataset with many correlated numerical features into a smaller set of new features called principal components. Each principal component is a weighted linear combination of the original variables and is constructed to capture as much variance (information) as possible. The first component captures the most variance, the second captures the next most while being orthogonal to the first, and so on. PCA helps reduce complexity, remove redundancy caused by correlations, and makes it easier to visualize patterns in 2D or 3D, while still retaining most of the original dataset’s information.

Important: PCA requires quantitative/numeric features only. Labels (team name, player name, wicket type, etc.) must be removed (not remapped to numbers).

Clustering

Compare & Contrast: KMeans vs Hierarchical vs DBSCAN

KMeans is a partition-based clustering algorithm that splits the dataset into k clusters by minimizing within-cluster variance (distance to cluster centroids). It works well when clusters are roughly spherical and similar in size, but requires choosing k in advance and can be sensitive to outliers.

Hierarchical clustering builds a tree of clusters (dendrogram) either by merging points bottom-up (agglomerative) or splitting top-down (divisive). It does not require selecting k initially; instead, you choose a cut level on the dendrogram. It can capture nested structure but may be slower on large datasets.

DBSCAN is a density-based algorithm that groups points in dense regions and labels isolated points as noise/outliers. It does not require choosing k, can find non-spherical clusters, and handles outliers naturally. However, it requires tuning eps and min_samples and may struggle when cluster densities vary a lot.

ARM (Association Rule Mining)

(a) Overview: What is ARM / Apriori?

Association Rule Mining (ARM) discovers “if-then” relationships in transaction-style datasets. A rule looks like {A, B} → {C}, meaning that when items A and B occur together, item C is likely to occur as well. Rules are evaluated using support (how often the rule appears in all transactions), confidence (how often C appears when A and B appear), and lift (how much more likely C is given A and B compared to C overall). The Apriori algorithm works by generating frequent itemsets (item combinations) using a minimum support threshold, then constructing rules from those frequent itemsets and filtering them by confidence and lift.

(b) Data Prep: Transaction Data Only (Unlabeled)

ARM requires transaction data, where each row represents a “basket” of items. In cricket terms, a transaction can be: (1) all events in an over, (2) events in an innings segment, or (3) simplified categorical events for a delivery. This dataset must be unlabeled and contain only item presence/absence (one-hot encoded). For example, a transaction could contain items like: {"powerplay", "boundary", "wide", "wicket"}.

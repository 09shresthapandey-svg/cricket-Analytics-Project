Cricket Analytics Project

Introduction

Cricket is one of the most widely followed sports in the world, with billions of viewers across countries such as India, England, Australia, Pakistan, and South Africa. The game is played in multiple formats including Test cricket, One Day Internationals (ODIs), and Twenty20 (T20), each demanding different skills, strategies, and performance expectations from players. Performance in cricket is influenced by a complex interaction of factors such as player roles, match format, pitch conditions, weather, opposition strength, and venue characteristics. Batting, bowling, and all-round performance are measured using a variety of statistics including runs scored, strike rate, economy rate, wickets taken, and match impact.

In recent years, the availability of large-scale cricket data has transformed how the sport is analyzed and understood. Advanced analytics are now used to evaluate player consistency, predict match outcomes, identify emerging talent, and optimize team composition. Public sources such as Cricsheet provide ball-by-ball data that enables deeper exploration of performance patterns. This project uses the Cricsheet T20 dataset to analyze scoring, wickets, teams, and players through exploratory data analysis and then extends the work using unsupervised learning techniques.

In Module 2, the project applies Principal Component Analysis (PCA) to reduce dimensionality and visualize patterns in 2D/3D, clustering algorithms (KMeans, Hierarchical, DBSCAN) to group similar player profiles, and Association Rule Mining (ARM) to discover “if-then” relationships between match events. These methods help transform complex ball-by-ball data into interpretable insights that can support selection, strategy, and performance evaluation.

PCA (Principal Component Analysis)

What is PCA?

Principal Component Analysis (PCA) is a dimensionality reduction technique that transforms a dataset with many numerical features into a smaller set of new variables called principal components. These components are linear combinations of the original features and are ordered so that the first captures the most variance (information), the second captures the next most while being orthogonal to the first, and so on. PCA helps reduce redundancy from correlated variables, simplify analysis, and makes it easier to visualize patterns in 2D or 3D.

Important (Requirement): PCA must use numeric features only. For this dataset, label columns such as team names, player names, and wicket_type were excluded from PCA.

Dataset Used for PCA

The cleaned dataset (combined_30_matches.csv) contains both numeric and label columns. For PCA, I used only the numeric performance-related features available in my cleaned file: innings, ball, runs_off_bat, and extras. Identifier and label columns such as match_id, team names, player names, and wicket type were removed.

Standardization

Before PCA, the features were standardized using StandardScaler so each variable has mean 0 and standard deviation 1. This is required because PCA is sensitive to scale, and cricket features can be measured in different ranges.

PCA Results (2D + 3D)

PCA was applied with n_components=2 and n_components=3 to visualize the data in 2D and 3D. I also computed (1) the percent variance retained in 2D and 3D, (2) the number of components required to retain at least 95% variance, and (3) the top three eigenvalues.

Clustering

Compare & Contrast: KMeans vs Hierarchical vs DBSCAN

KMeans is a centroid-based algorithm that partitions data into k clusters by minimizing within-cluster distances to centroids. It works best when clusters are roughly spherical and similar size, but requires selecting k and can be sensitive to outliers.

Hierarchical clustering creates a tree of clusters (dendrogram) that shows nested grouping structure. It does not require choosing k upfront and can reveal multi-level cluster structure, but can be slower on large datasets.

DBSCAN is density-based and groups points in dense regions while labeling sparse points as noise. It does not require selecting k and can find non-spherical clusters, but requires tuning parameters (eps, min_samples) and can struggle if densities vary.

ARM (Association Rule Mining)

What is ARM / Apriori?

Association Rule Mining (ARM) discovers “if-then” relationships in transaction data. A rule looks like {A, B} → {C}, meaning when A and B occur together, C is likely to occur. Rules are evaluated using: support (frequency of the rule in all transactions), confidence (probability of C given A,B), and lift (strength of association relative to chance). The Apriori algorithm finds frequent itemsets above a support threshold and generates association rules from them.

Transaction Dataset (How it was created)

ARM requires transaction-style data rather than raw ball-by-ball rows. Using the cleaned dataset, I defined each over as a transaction (grouped by match_id, innings, and over number derived from ball). For each over, I generated categorical items such as: WICKET, BOUNDARY, EXTRAS, HIGH_RUN_OVER, and DOT_HEAVY. This produces an unlabeled one-hot encoded transaction matrix suitable for Apriori.

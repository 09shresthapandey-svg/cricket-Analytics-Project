Cricket Analytics Project

Introduction

This project analyzes T20 cricket ball-by-ball data from Cricsheet using a full data science workflow. It includes data cleaning, exploratory data analysis (EDA), Principal Component Analysis (PCA), clustering (KMeans, Hierarchical, DBSCAN), and Association Rule Mining (ARM).

Cricket performance can be studied using ball-by-ball data, which captures scoring events, extras, and wicket outcomes at a granular level. Using this dataset, we can explore scoring patterns, team/player contributions, and match dynamics using modern data analysis methods.

In Module 2, PCA helps reduce dimensionality and visualize numeric structure, clustering groups similar player profiles, and ARM discovers if-then relationships between over-level events (e.g., wickets, boundaries, high-run overs).

PCA (Principal Component Analysis)

PCA Description (1 paragraph)

Principal Component Analysis (PCA) is a dimensionality reduction technique that transforms correlated numeric variables into a smaller set of uncorrelated variables called principal components. Each component is a weighted linear combination of the original features, and the components are ordered so that the first captures the most variance (information), followed by the second, third, and so on. PCA is useful for visualization (2D/3D plots), identifying which variables contribute most to variation, and simplifying the dataset while retaining most information.

Dataset Choice + Links

Clustering

Compare and Contrast: KMeans vs Hierarchical vs DBSCAN

KMeans is a partition-based method that assigns points to k clusters by minimizing within-cluster variance (distance to centroids). It is fast and works well when clusters are roughly spherical, but requires choosing k in advance and is sensitive to scaling and outliers. Hierarchical clustering builds a tree (dendrogram) that shows nested cluster structure and does not require choosing k at the start; however, it can be slower on large datasets and merges/splits are not reversible. DBSCAN is density-based: it groups points in dense regions and labels sparse points as noise. DBSCAN can find non-spherical clusters and naturally detect outliers, but requires good choices of eps and min_samples.

Data Prep (required): I created a labeled dataset where each row corresponds to a player (striker) and the remaining columns are quantitative player-profile features. I saved the label column separately, removed it before clustering, ensured the remaining data was numeric-only, and normalized using StandardScaler.

Choosing k using the Silhouette Method

I computed silhouette scores across multiple k values. Higher silhouette scores indicate better-separated clusters. Based on these results, the top three k values were: k = 2, k = 3, and k = 7.

Hierarchical Clustering

Compared to KMeans, hierarchical clustering provides a nested view of group structure. Cutting the dendrogram at an appropriate height produces clusters that can be compared to the KMeans groupings.

DBSCAN

DBSCAN identifies dense player groups and labels outliers as noise. Compared to KMeans, DBSCAN does not force every player into a cluster and can highlight unusual profiles.

Overall Interpretation (required discussion)

KMeans with k = 2 and k = 3 produced broad, interpretable player groupings, while k = 7 created more granular sub-groups. Hierarchical clustering supported similar grouping patterns and clarified how clusters relate. DBSCAN additionally identified possible outliers (noise points) that KMeans would otherwise force into a cluster. Among the tested values, k = 3 provides a strong balance between interpretability and separation based on silhouette results.

Association Rule Mining (ARM)

(a) Overview

Association Rule Mining (ARM) discovers “if-then” relationships in transaction data (e.g., If A happens, B tends to happen). A rule is written as A → B. Support is the proportion of transactions that contain both A and B (how frequent the rule is overall). Confidence is the probability that B occurs given A occurs (how reliable the rule is). Lift compares confidence to the baseline probability of B; lift > 1 indicates a positive association beyond chance. The Apriori algorithm finds frequent itemsets above a minimum support threshold and then generates rules that meet minimum confidence (and optionally lift), using the idea that all subsets of a frequent itemset must also be frequent.

(b) Data Prep

ARM requires unlabeled transaction data, where each row is a “basket” of items (no class labels). In this project, I treated each over as a transaction (grouped by match_id, innings, and over number derived from ball). Each transaction contains items representing events such as: WICKET, BOUNDARY, EXTRAS, DOT_HEAVY, and HIGH_RUN_OVER. Below is a screenshot of the transaction-format dataset used for Apriori.
